### The Problem With How We Prompt

Most of us treat AI like Google. We type short questions and hope for the best:

* “How do I configure OSPF?”
* “Fix this config.”
* “What’s wrong with my routing table?”

And then we wonder why the responses are generic, outdated, or simply incorrect.

Here’s what nobody tells you: **AI isn’t a search engine. It’s a prediction engine.**

Think about it like this. When you’re training a junior engineer, you don’t just say “configure the router” and walk away. You give them context: which router, what’s the current state, what’s the end goal, what conventions does your team follow?

AI needs the same treatment. The difference? AI won’t ask follow-up questions or make reasonable assumptions based on your company’s standards. It’ll just… guess. And those guesses often suck.

### The Secret: Treat Prompts Like Programs

This is the mindset shift that changed everything for me.

A prompt isn’t a question — **it’s a set of instructions you’re programming into the AI.**

Just like a Python script or a network automation playbook, your prompt needs:

* Clear inputs (context)
* Defined behavior (persona)
* Specific outputs (format)

Let me show you what this looks like in practice.

### The Four Elements Every Good Prompt Needs

### 1. Persona: Give the AI an Identity

Instead of letting the AI be a generic “helpful assistant,” assign it a specific role.

**Bad prompt:**

```
Help me with network automation
```

**Good prompt:**

```
You are a network automation engineer with expertise in Python and network APIs.
```

Why does this work? Because it narrows the AI’s focus. Instead of pulling from every random tutorial ever written, it responds as an actual automation engineer would.

**Real example from my work:**

I needed to write a Python script to pull interface statistics from multiple switches. Instead of asking “how do I get interface stats,” I tried:

```
You are a network automation engineer specializing in Python and Netmiko.  
I need to collect interface error counters from 20 Cisco switches and export to CSV.  
Requirements:  
- Use Netmiko for SSH connections  
- Collect input/output errors and CRC errors  
- Handle connection timeouts gracefully  
- Format: hostname, interface, input_errors, output_errors, CRC  
Write a Python script that accomplishes this.
```

The response included proper error handling and CSV formatting. Saved me an hour of piecing together Stack Overflow answers.

```
#!/usr/bin/env python3  
import csv  
import re  
import time  
from datetime import datetime  
from getpass import getpass  
from netmiko import ConnectHandler, NetmikoTimeoutException, NetmikoAuthenticationException  
  
def get_interface_errors(device):  
    """  
    Connect to a Cisco device and collect interface error counters.  
      
    Args:  
        device (dict): Dictionary containing device connection parameters  
          
    Returns:  
        list: List of dictionaries with interface error data  
    """  
    interface_errors = []  
      
    try:  
        # Establish SSH connection  
        print(f"Connecting to {device['host']}...")  
        net_connect = ConnectHandler(**device)  
          
        # Get interface information  
        output = net_connect.send_command("show interfaces")  
          
        # Parse the output to extract error counters  
        interfaces = re.split(r'^(?=\S+\s+is\s+.+)', output, flags=re.MULTILINE)[1:]  
          
        for interface in interfaces:  
            # Extract interface name  
            interface_match = re.search(r'^(\S+)\s+is\s+.+', interface, re.MULTILINE)  
            if not interface_match:  
                continue  
            interface_name = interface_match.group(1)  
              
            # Skip VLAN interfaces  
            if interface_name.startswith('Vlan'):  
                continue  
                  
            # Extract input errors  
            input_errors_match = re.search(r'(\d+)\s+input errors', interface)  
            input_errors = input_errors_match.group(1) if input_errors_match else "0"  
              
            # Extract output errors  
            output_errors_match = re.search(r'(\d+)\s+output errors', interface)  
            output_errors = output_errors_match.group(1) if output_errors_match else "0"  
              
            # Extract CRC errors  
            crc_match = re.search(r'(\d+)\s+CRC', interface)  
            crc_errors = crc_match.group(1) if crc_match else "0"  
              
            # Store the interface data  
            interface_errors.append({  
                'hostname': device['host'],  
                'interface': interface_name,  
                'input_errors': input_errors,  
                'output_errors': output_errors,  
                'crc_errors': crc_errors  
            })  
          
        # Disconnect from the device  
        net_connect.disconnect()  
        print(f"Successfully collected data from {device['host']}")  
        return interface_errors  
          
    except NetmikoTimeoutException:  
        print(f"Connection timeout to {device['host']}")  
        return []  
    except NetmikoAuthenticationException:  
        print(f"Authentication failed for {device['host']}")  
        return []  
    except Exception as e:  
        print(f"Error connecting to {device['host']}: {str(e)}")  
        return []  
  
def main():  
    # Get device credentials  
    username = input("Enter username: ")  
    password = getpass("Enter password: ")  
      
    # Define the list of switches (replace with your actual switch IPs/hostnames)  
    switch_ips = [  
        "192.168.1.1",  
        "192.168.1.2",  
        "192.168.1.3",  
        "192.168.1.4",  
        "192.168.1.5",  
        "192.168.1.6",  
        "192.168.1.7",  
        "192.168.1.8",  
        "192.168.1.9",  
        "192.168.1.10",  
        "192.168.1.11",  
        "192.168.1.12",  
        "192.168.1.13",  
        "192.168.1.14",  
        "192.168.1.15",  
        "192.168.1.16",  
        "192.168.1.17",  
        "192.168.1.18",  
        "192.168.1.19",  
        "192.168.1.20"  
    ]  
      
    # Prepare device connection parameters  
    devices = []  
    for ip in switch_ips:  
        devices.append({  
            'device_type': 'cisco_ios',  
            'host': ip,  
            'username': username,  
            'password': password,  
            'timeout': 10,  # Set timeout to 10 seconds  
            'session_timeout': 60,  
        })  
      
    # Collect interface errors from all devices  
    all_interface_errors = []  
    for device in devices:  
        interface_errors = get_interface_errors(device)  
        all_interface_errors.extend(interface_errors)  
        # Add a small delay between connections to avoid overwhelming the network  
        time.sleep(1)  
      
    # Generate output filename with timestamp  
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")  
    filename = f"interface_errors_{timestamp}.csv"  
      
    # Write the collected data to CSV  
    if all_interface_errors:  
        with open(filename, 'w', newline='') as csvfile:  
            fieldnames = ['hostname', 'interface', 'input_errors', 'output_errors', 'crc_errors']  
            writer = csv.DictWriter(csvfile, fieldnames=fieldnames)  
              
            writer.writeheader()  
            for error_data in all_interface_errors:  
                writer.writerow(error_data)  
                  
        print(f"Data exported to {filename}")  
    else:  
        print("No data collected. CSV file not created.")  
  
if __name__ == "__main__":  
    main()
```

#### Features

* **Connection Handling:** The script gracefully handles connection timeouts and authentication failures
* **Error Parsing:** Extracts input errors, output errors, and CRC errors from interface statistics
* **CSV Export:** Creates a timestamped CSV file with all collected data
* **Progress Feedback:** Prints status messages during execution
* **Rate Limiting:** Includes a small delay between connections to avoid overwhelming the network

### 2. Context: Stop Making the AI Guess

This is where most network engineers (myself included) mess up. We assume the AI “knows” things it doesn’t.

**Bad prompt:**

```
Configure QoS for VoIP
```

**Good prompt:**

```
I'm working on a Cisco Catalyst switch running IOS 15.2.  
Need to configure QoS for VoIP traffic:  
- VoIP uses DSCP EF (46)  
- Should get priority queuing  
- Interfaces: Gi1/0/1 through Gi1/0/10  
- Bandwidth guarantee: 30% of link capacity  
Generate the complete configuration.
```

See the difference? The second prompt gives the AI everything it needs to generate accurate, platform-specific commands.

```
! -------------------------------------------------  
! Step 0: Enable QoS Globally on the Catalyst Switch  
! This is a REQUIRED first step on most Catalyst switches.  
! Without this command, no QoS policies will be active.  
! -------------------------------------------------  
mls qos  
  
! -------------------------------------------------  
! Step 1: Define the class of traffic (VoIP)  
! This class-map matches any traffic already marked   
! with DSCP Expedited Forwarding (EF), which is 46.  
! -------------------------------------------------  
class-map match-any VOIP-TRAFFIC  
 description "Matches VoIP traffic marked with DSCP EF (46)"  
 match dscp ef  
  
! -------------------------------------------------  
! Step 2: Define the QoS policy  
! This policy-map creates a strict priority queue for   
! the VOIP-TRAFFIC class and guarantees it 30% of the  
! interface bandwidth. All other traffic falls into   
! the class-default queue.  
! -------------------------------------------------  
policy-map VOIP-POLICY  
 description "QoS Policy for VoIP Traffic"  
  class VOIP-TRAFFIC  
   priority percent 30  
  class class-default  
   fair-queue  
  
! -------------------------------------------------  
! Step 3: Apply the policy to the target interfaces  
! The service-policy is applied in the OUTPUT direction,  
! as this controls how traffic is queued and transmitted  
! out of the interface towards the end device (e.g., IP phone).  
! -------------------------------------------------  
interface range GigabitEthernet1/0/1 - 10  
 description "Access Ports for VoIP Devices"  
 service-policy output VOIP-POLICY
```

**Pro tip:** When it comes to context, over-sharing is better than under-sharing. Include:

* Device platform and OS version
* Current state vs. desired state
* Constraints or requirements
* Any error messages or logs

### 3. Format: Tell the AI How to Respond

By default, AI will ramble. A lot.

If you want concise, actionable output, you need to specify exactly what format you want.

**Examples:**

For quick answers:

```
Respond in 3 bullet points, maximum 30 words each.
```

For configs:

```
Provide only the CLI commands, no explanations. Format for copy-paste directly into the device.
```

```
configure terminal  
mls qos  
class-map match-any VOIP-TRAFFIC  
 description "Matches VoIP traffic marked with DSCP EF (46)"  
 match dscp ef  
policy-map VOIP-POLICY  
 description "QoS Policy for VoIP Traffic"  
  class VOIP-TRAFFIC  
   priority percent 30  
  class class-default  
   fair-queue  
interface range GigabitEthernet1/0/1 - 10  
 description "Access Ports for VoIP Devices"  
 service-policy output VOIP-POLICY  
end
```

For documentation:

```
Write this as a step-by-step runbook with numbered steps. Include verification commands after each configuration step.
```

I use this constantly when creating network documentation. Instead of getting long-winded explanations, I get clean, structured docs I can actually use.

### 4. Examples: Show, Don’t Tell (Few-Shot Prompting)

This technique is a *chef’s kiss* for consistency.

Instead of describing what you want, show the AI examples of your desired output.

**Real scenario:** I needed to generate standardized device naming conventions for our inventory system.

Instead of describing our format, I did this:

```
Generate device names following this exact pattern:  
  
Example 1: NYC-RTR-CORE-01 (New York, Router, Core layer, sequence 01)  
Example 2: LAX-SWH-ACCESS-12 (Los Angeles, Switch, Access layer, sequence 12)  
Example 3: CHI-FWL-EDGE-03 (Chicago, Firewall, Edge, sequence 03)  
  
Now create names for:  
- Dallas distribution switch, sequence 05  
- Seattle access switch, sequence 23  
- Boston core router, sequence 02
```

The AI nailed the format perfectly. No back-and-forth, no adjustments needed.

```
Based on the naming pattern provided, here are the device names for your requested equipment:  
  
- DAL-SWH-DISTRIBUTION-05 (Dallas, Switch, Distribution layer, sequence 05)  
- SEA-SWH-ACCESS-23 (Seattle, Switch, Access layer, sequence 23)  
- BOS-RTR-CORE-02 (Boston, Router, Core layer, sequence 02)
```

### Advanced Techniques (When You Need to Level Up)

### Chain of Thought: Make the AI Show Its Work

For complex troubleshooting, ask the AI to think step-by-step before providing an answer.

```
Network issue: Users on VLAN 20 can't reach servers on VLAN 30, but VLAN 10 can reach both.  
  
Before providing your answer, think through this step-by-step:  
1. What components are involved in inter-VLAN routing?  
2. What could cause selective VLAN connectivity issues?  
3. What are the most likely configuration problems?  
  
Then give me a prioritized troubleshooting checklist.
```

This gives you two benefits:

* You see the AI’s reasoning (helps you catch mistakes)
* The final answer is usually more accurate

### Tree of Thought: Explore Multiple Solutions

When you need options, not just one answer:

```
I need to segment a flat network for better security and traffic management.  
Generate 3 different approaches:  
1. VLAN-based segmentation with ACLs  
2. VRF-lite for logical separation  
3. Micro-segmentation with firewall zones  
For each, explain implementation complexity, scalability, and management overhead. Then recommend the best option for a 500-user environment.
```

This is perfect for design decisions where there’s no single “right” answer.

### The Playoff Method: AI Debates Itself

This sounds weird but works incredibly well for complex decisions.

```
Scenario: Should we upgrade our entire network to support 10Gbps or implement targeted upgrades only where needed?  
  
Persona 1 (Infrastructure advocate): Make the strongest case for full network upgrade  
Persona 2 (Budget analyst): Make the strongest case for targeted upgrades only  
Persona 3 (Network architect): Evaluate both arguments and provide a balanced recommendation  
  
Context: 200-node network, current infrastructure is 1Gbps, some bandwidth congestion during peak hours, limited budget for next fiscal year.
```

You get a balanced view with all angles considered. Way better than a simple “which is better?” question.

### Putting It All Together: A Template You Can Use

Here’s the framework I use for most of my network engineering prompts:

```
# PERSONA  
You are [specific role with relevant expertise]  
# CONTEXT  
- Current situation: [what exists now]  
- Environment details: [devices, versions, topology]  
- Constraints: [budget, time, compatibility requirements]  
# TASK  
[Exactly what you need the AI to do]  
# FORMAT  
- [How you want the output structured]  
- [Any specific requirements]  
# EXAMPLES (if applicable)  
[Show 1-2 examples of desired output]
```

**Example in action:**

```
# PERSONA  
You are a network engineer specializing in wireless network design  
# CONTEXT  
- Environment: 3-floor office building, 15,000 sq ft per floor  
- Current: Single SSID, consumer-grade APs, frequent disconnections  
- Users: 200 employees with laptops and mobile devices  
- Constraint: Must use existing Cisco infrastructure  
# TASK  
Design a wireless network architecture that provides reliable coverage and separates guest traffic  
# FORMAT  
- High-level design diagram description  
- AP placement strategy  
- SSID configuration recommendations  
- Security recommendations for each SSID  
# EXAMPLES  
Similar to a typical enterprise design:  
- Corporate SSID: WPA3-Enterprise with 802.1X  
- Guest SSID: Captive portal with VLAN isolation  
- IoT SSID: PSK with limited network access
```

### The One Thing That Matters Most

Here’s the brutal truth: **all these techniques won’t help if you don’t know what you actually want.**

The real skill isn’t memorizing prompt templates. It’s **clarity of thought**.

Before you type anything into ChatGPT or Claude, take 30 seconds and ask yourself:

* What specific problem am I trying to solve?
* What information does the AI need to help me?
* What does a successful answer look like?

If you can’t articulate this to yourself, the AI definitely can’t figure it out.

This is actually the same skill that makes you better at:

* Writing clear documentation
* Delegating to junior engineers
* Explaining issues to management
* Debugging complex problems

Prompt engineering is just thinking clearly, then communicating that clarity to a machine.

### Start Simple, Then Iterate

You don’t need to write perfect prompts on the first try.

My workflow:

1. Start with a basic prompt (persona + context + task)
2. See what the AI returns
3. Adjust the prompt based on what was wrong
4. Save the good version as a template

Over time, you’ll build a library of prompts that work for your specific needs:

* Troubleshooting templates
* Config generation formats
* Documentation structures
* Design review frameworks

### Real Talk: This Actually Saves Time

I was skeptical at first. Writing longer prompts feels slower than just asking quick questions.

But here’s what I found after a month:

**Before:**

* Ask vague question → Get generic answer → Rephrase → Get closer answer → Spend 10 minutes refining → Maybe get something useful

**After:**

* Spend 2 minutes writing a clear prompt → Get exactly what I need on first try

The upfront investment pays off immediately. And the more you do it, the faster you get at writing good prompts.

Next time you’re about to ask an AI tool something, stop.

Spend one extra minute adding:

* A persona (what role should the AI play?)

Just those three things. See what happens.

I bet you’ll be surprised at how much better the response is.

### Quick Reference Card

Keep this handy:

**Basic Template:**

```
You are [role].  
Context:  
- [Key detail 1]  
- [Key detail 2]  
- [Key detail 3]  
Task: [What you need]  
Format: [How you want it]
```

**Common Mistakes:**

* ❌ “Configure ACL” → ✅ Include device platform, traffic requirements, and specific permit/deny rules
* ❌ “Debug network issue” → ✅ Specify symptoms, affected devices, topology, and error messages
* ❌ No format specified → ✅ Tell it exactly how to structure the response
* ❌ Assuming context → ✅ Over-explain rather than under-explain

**When to Use Advanced Techniques:**

* **Chain of Thought:** Complex troubleshooting, need to see reasoning
* **Tree of Thought:** Design decisions need multiple options
* **Few-Shot:** Consistency matters, specific format required
* **Playoff Method:** High-stakes decisions, want a balanced perspective
