### Docker Networking Basics — Default Bridge Network

#### Part — 01

In the world of tech and software, Docker is a cool and powerful tool. It uses something called containers that work well and superfast. But there’s one tricky part — that is Docker networking. Normally, when you use Docker, it connects things without you even noticing. But, how does it do that? In this series of blogs, these are the things we’re going to explore to understand Docker networking, so grab your coffee and dive in!

Let’s give you the tools and knowledge to change how you think about Docker containers, there are a few things you need to do first.

#### **Docker Host — Linux Virtual Machine**

We’ll use a virtual machine that runs on VMware/VirtualBox. In my case, I’m using Ubuntu Desktop virtual machine, a free and versatile software.

#### **Docker’s Default Bridge Network**

If you haven’t installed Docker on your computer yet, don’t worry. I want to show you something important before we move on. Let’s open the command prompt together because we’re about to do something cool.

Type the command `ip address show`. It will show you two things – your computer's internal connection (loopback) and the main one that connects you to your home network.


Now, let’s get to — installing Docker. We’ll begin with a simple `sudo apt update`to ensure our repositories are up-to-date, following that, a `sudo apt install docker.io -y` is all it takes.

If you return your attention to the terminal and execute `ip addr show`once more, you’ll notice something new, ***Docker zero*** has appeared.

```
3: docker0: &lt;NO-CARRIER,BROADCAST,MULTICAST,UP&gt; mtu 1500 qdisc noqueue state DOWN group default
```

***Docker zero*** is our brand-new virtual interface, the default interface and network for Docker — your gateway to Docker networking, pay close attention to its IP address, you’ll need it later.

But wait, there’s more to uncover. Let’s run the command, `docker network ls` and you’ll see a list of our current Docker networks.


One of them we’re already familiar with: is **Bridge**. But there are two more, and we’ll explore those in the next blogs. These networks are part of the seven default Docker networks.

Let’s deploy some containers within our default network. We’ll use the`docker run`command with a few switches to make it interactive and detached, along with the `— rm`switch to ensure it cleans up after stopping itself.

First, we’ll name our container, and for this one, we’ll call it Tor, and finally, we’ll specify the image; in this case, I prefer the quick, lightweight, and fast “**busy box**” image.

```
zolo@ubuntu-vm:~$ sudo docker run -itd --rm --name tor busybox  
2ac6db9c537f8dbb5d9178ba34dd156778064ba5d097a99adf68daa299480565
```

Now, let’s add another container. Press the up arrow key, and change the name from **Tor** to **Troy**, you have now, two containers up and running.

```

```

But we’re not done yet, our final act is, to introduce a **nginx** container, naming it **Stone**.

```

998d4401d287576754ed6395fa954129222e13484909616e67c2dfc4f1a0e579
```

The stage is set for an exciting and fast-paced journey through the world of Docker networking.

#### Containers Deployed on the Default Bridge Network

We deploy containers without explicitly specifying any networking, allowing Docker to work in its default. Let’s confirm that our containers are up and running by using `docker ps`. As you can see, they are all present and ready to go.

The ***Docker zero,*** bridge network is a virtual switch that connects containers effortlessly. Each container has its connection, letting them communicate within the Docker network. It’s all happening behind the scenes, making Docker containers truly magical.

#### Docker’s Virtual Interfaces and DHCP

As we explore the inner workings of Docker networking, let’s focus on the remarkable feats that Docker accomplishes behind the scenes. When containers are deployed, Docker takes care of creating virtual interfaces and assigning IP addresses within the bridge network.

Each of our containers is given a *veth* interface, connecting them within the Docker bridge network, let's see it in action. When we execute `ip addr show`again, we find three new interfaces.

#### Inter-Container Communication and Internet Access in Docker

One of the enchanting aspects of Docker networking is how it allows containers to seamlessly communicate with each other and access the internet using the same DNS and network, where containers can ping the internet effortlessly.

The secret to this magic lies in how Docker handles DNS. Docker takes a copy of the *resolv.conf* file from the host (Ubuntu) and place it inside each container. This means that all containers share the same DNS, ensuring consistent and reliable name resolution. But that’s just the beginning.

Thanks to the ***Docker zero*** network, which acts like a virtual switch, containers can communicate with each other effortlessly, let’s see the containers in action. We can use the `docker exec`command, followed by the *-it* flag and the name of the container we want to explore.

For this demonstration, let’s start with **Tor**. Once inside the container’s shell, you can execute `ip addr show`to view its IP address.

#### Exposing Docker Container Ports for Website Access

Now, you might expect that you could simply navigate (from the local PC) to the IP address of the host (Ubuntu) on port 80 and access the website hosted by the Docker container. But, alas, it’s not that simple.

As you probably guessed, trying *192.168.8.101:80* won’t get you anywhere, and this is one of the quirks of the bridge network.

By default, Docker doesn’t allow access to services offered by containers like websites. It’s one of those little distinctivenesses, while necessary for security, can sometimes be frustrating.

But there’s a solution, you can manually expose the ports of your Docker containers to access the services they offer. This process is something you might have done before, and if not, let’s explore it together right now.

To make **Stone** accessible to the world, we need to take a simple but crucial step — exposing port *80*. This allows us to access our web server and experience its magic.

First, let’s gracefully exit the container, now, let’s remove the stone. Docker makes it easy with a simple command: `docker stop`. The container will stop gracefully, and Docker will remove it automatically.

```
zolo@ubuntu-vm:~$ sudo docker stop stone  
stone
```

Now — redeploying the stone. But don’t worry, it’s nothing too complicated. Instead of typing the command from scratch, we can make it even quicker. Just press the *up* arrow key, and you’ll see the previous command reappear. Now, right before specifying the container name, add a *-p* followed by *80:80*. This tells Docker to expose port *80* from the container and map it to port *80* on your host.

```
zolo@ubuntu-vm:~$ sudo docker run -itd -p 80:80 --rm --name stone nginx  
fe2e9f22c6df77852a0584db5058b367d86c67baf35cc8ab11f2bc62e692a1c0
```

With this small but impactful change, stone’s web services will be accessible to the world. It’s a simple yet powerful demonstration of how Docker allows you to control the accessibility of your containers, turning them into dynamic and versatile components of your development and hosting environment.

Simply type `docker ps`, and it will display a list of your running containers along with the exposed ports. This is a handy way to keep track of which ports are being exposed by your containers. It’s like your Docker dashboard, helping you maintain control over your containerized services.

```
zolo@ubuntu-vm:~$ sudo docker ps  
CONTAINER ID   IMAGE     COMMAND                  CREATED          STATUS          PORTS                               NAMES  
fe2e9f22c6df   nginx     "/docker-entrypoint.…"   8 seconds ago    Up 6 seconds    0.0.0.0:80-&gt;80/tcp, :::80-&gt;80/tcp   stone  
1948a3792af1   busybox   "sh"                     42 minutes ago   Up 42 minutes                                       troy  
2ac6db9c537f   busybox   "sh"                     43 minutes ago   Up 43 minutes                                       tor
```

Now, refresh your web page to see the changes in action. The web server is up and running, ready to serve its magic to the world.

While exposing ports might seem like a routine task, it’s a good practice. It introduces a layer of isolation between your containers and your network, enhancing security and control.

With the ability to expose and manage ports, Docker empowers you to fine-tune the accessibility of your containers. It’s all about finding the right balance between convenience and security. 👍
