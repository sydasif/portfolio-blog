### Why Learn Basic Terminal Commands — Even If You’re Working with AI

Your AI assistant suggests running this command while troubleshooting a network issue:

```
find /var/log -name "*.log" | grep "error"
```

You **pause** for a second. Should you press **Enter**? What will this actually do to your system?

That hesitation is common. Many network engineers now work with AI tools that suggest or even execute terminal commands. The risk is not AI itself. The risk is approving commands you cannot read with confidence.

This guide fixes that problem. You will learn how to **understand**, **judge**, and **safely run** the terminal commands AI tools use every day — without turning into a Linux expert.

---

### The One Rule That Makes the Terminal Simple

Every terminal command does **one of three things**:

1. **Read** data
2. **Write** data
3. **Delete** data

Once you see commands this way, fear disappears, and judgment improves.

---

#### The Terminal: A Text Interface You Already Know

You already use this model on network devices.

* GUI for convenience
* CLI for control

Your computer works the same way. The terminal is a text interface that sends instructions to the operating system.

When you open Terminal (macOS), PowerShell or WSL (Windows), or a Linux shell, you are using a **shell**. The most common one is **Bash**.

Bash does not protect you. There are no confirmation pop-ups. You type a command, press Enter, and it runs.

That is power. That is also a responsibility.

---

### Getting Started

Open a terminal:

* **macOS**: Cmd + Space → Terminal
* **Windows**: PowerShell or WSL
* **Linux**: Ctrl + Alt + T

You will see something like:

```
user@hostname:~$
```

* `$` means regular user
* `~` means home directory

Run this:

```
ls    # read
```

You just asked the system to **show files**. Nothing changed.

#### How Commands Actually Execute

Bash itself knows very little. It mostly launches other programs.

When you type:

```
git status
```

Bash finds the `git` program and passes `status` to it. Bash is just the dispatcher.

This matters because AI tools rely on this same mechanism. They do not have special powers. They use the same shell you do.

### Commands AI Tools Use Most Often

#### Looking Around (Read-Only)

These commands inspect your system. They do not modify anything.

```
ls                     # read  
ls -la                 # read (detailed)  
pwd                    # read  
cat file.txt           # read  
head -n 20 log.txt     # read  
tail -f syslog         # read (live)
```

#### Searching Files and Configs (Read-Only)

AI tools spend a lot of time searching.

```
find . -name "*.yaml"          # read  
grep "interface" config.txt   # read  
grep -r "192.168" /etc/       # read
```

Fast search tools:

```
rg "vlan" --type yaml          # read  
rg "password" --hidden        # read
```

#### Running Automation and Tools (Write)

These commands **change state**.

```
pip install netmiko           # write  
python backup.py              # write  
git commit -m "Update config" # write  
ansible-playbook site.yaml    # write
```

Before approving these, ask one question:

**What will this change if it succeeds?**

#### Creating, Moving, and Deleting Files

```
mkdir configs/                # write  
touch new.yaml                # write  
cp base.yaml sw1.yaml         # write  
mv old.py new.py              # write  
rm temp.log                   # delete  
rm -r old_configs/            # delete
```

Deletion is permanent. No recycle bin.

#### Understanding Pipes and Chaining

The pipe `|` passes output from one command to another.

```
cat system.log | grep "fail"
```

Translation:  
 *“Show the file, then filter lines that contain ‘fail’.”*

Conditional execution:

```
ping 8.8.8.8 && echo "Link works"
```

The second command runs only if the first succeeds.

#### Reading a Complex Command With Confidence

Command:

```
find /var/log -type f -name "*.log" -mtime -7 | xargs grep "connection refused"
```

Break it down:

* `find /var/log` → start here
* `-type f` → files only
* `-name "*.log"` → log files
* `-mtime -7` → last 7 days
* `xargs grep ...` → search inside them

Meaning:

**Search recent log files for connection refused errors.**

Now you can decide if it fits your intent.

#### A Simple AI Safety Rule

AI does not know your system.

It predicts what *usually* works.

You approve what *actually* runs.

Never treat AI commands as trusted by default.

### Red Flags That Deserve a Pause

**Force deletion with variables**

```
rm -rf $DIR/*
```

If `$DIR` expands wrong, damage follows.

**Running as admin without a reason**

```
sudo rm logs/
```

Ask why elevated rights are needed.

**Permission changes without scope**

```
chmod -R 777 scripts/
```

This weakens system safety.

**Executing remote code**

```
curl site/script.sh | bash
```

You just ran unknown code.

#### The Safest Trick: Dry Runs With echo

Before risky commands:

```
echo rm -rf $DIR/*
```

This shows what the shell expands **without executing it**.

If the output looks wrong, stop.

### When Commands Fail

Read the error text.

Common ones:

* `command not found` → tool missing
* `permission denied` → rights issue
* `no such file` → wrong path

Useful checks:

```
pwd  
ls
```

Paste full errors back into your AI tool. Debugging works best with exact output.

---

### Quick Risk Guide

**Read**

* `ls`, `cat`, `find`, `grep`, `rg`, `diff`

**Write**

* `mkdir`, `touch`, `cp`, `mv`, `pip install`, `git commit`

**Delete**

* `rm`, `rm -r`

**High Risk**

* `rm -rf`
* `sudo`
* `chmod`, `chown`

---

### Moving Forward

The terminal is not magic. It is just text instructions.

AI tools make suggestions.  
 You remain responsible for execution.

You don't need extensive Bash knowledge. You need **awareness**.

**Final rule to remember:**

&gt; *Never run a command you cannot explain in one sentence.*
